## 문제 진단

현재 `generatePrescription` 서버 함수는 `google/gemini-2.5-flash` 모델에 학교 진단 데이터만 넘기고 "선호 도구의 ○○기능을 정확히 지칭해"라고 강제합니다. 그러나 모델은 실제 한국 에듀테크 제품(옥수수, 똑똑수학탐험대, 칸아카데미, 뤼튼, 클래스팅 AI, 하이러닝 등)의 기능 명세를 정확히 모르기 때문에, "옥수수의 AI 기반 글쓰기"처럼 **실제로 존재하지 않는 기능을 자신 있게 만들어내는 할루시네이션**이 발생합니다. 교과 단원 예시(국어 논설문, 수학 도형 등)도 같은 방식으로 그럴듯하게 지어냅니다.

## 개선 방향 (4단계)

### 1. 도구 사실 시트(Tool Factsheet) 도입 — 핵심

`src/lib/tool-factsheet.ts` 신설. 자주 등장하는 한국 에듀테크 도구의 **검증된 사실 카드** 작성:

```text
옥수수(LXper):
  카테고리: 초·중등 수학 코스웨어
  핵심 기능: 학년·단원별 문항 자동 출제, AI 진단 리포트, 오답 클리닉
  없는 기능: 글쓰기 생성, 이미지 생성, 챗봇
똑똑수학탐험대: 초1-2 수학 게임형 진단·학습…
칸아카데미: …
뤼튼: 글쓰기·검색 챗봇 (수학 풀이 신뢰도 낮음)
하이러닝: …
클래스팅 AI: …
패들렛: …
미리캔버스: …
```

서버 함수에서 입력된 `tools` 중 시트에 있는 항목의 카드만 프롬프트에 동적 삽입. 시트에 없는 도구는 "기본 기능만 사용. 가상 기능 금지" 표시로 함께 전달.

### 2. 시스템 프롬프트 강화 — 할루시네이션 차단 룰 추가

`src/lib/prescription.functions.ts`의 `SYSTEM_PROMPT`에 다음 절을 추가:

- **사실성 규칙**: "도구 사실 시트(Tool Factsheet)에 명시된 기능만 사용하라. 시트에 없는 기능을 만들어내면 안 된다. 도구 카테고리(수학 코스웨어/글쓰기 챗봇 등)와 맞지 않는 활용은 절대 제안하지 마라."
- **단원 예시 가드레일**: "교육과정 단원명을 모르면 '학년 수준에 맞는 단원'처럼 일반화 표현을 사용하라. 가짜 단원명/성취기준 코드를 만들지 마라."
- **불확실 표시**: 각 평가 포인트·함정에 자체 근거가 약하면 "(현장 검증 권장)" 꼬리표를 붙이도록 유도.
- 기존 "두루뭉술하게 쓰지 말고 도구의 ○○기능을 정확히 지칭" 지시는 **시트에 있는 도구에 한해서만** 적용하도록 조건부로 수정.

### 3. 출력 스키마에 `assumptions` 필드 추가 + 사후 검증

`PrescriptionOutput`에 `assumptions: string[]` 추가 — 모델이 처방을 만들 때 사용한 도구 기능·단원 가정을 평문으로 명시.

서버 함수에서 응답을 받은 뒤 가벼운 검증 통과시킴:
- `assumptions` 또는 본문이 시트에 없는 "AI 글쓰기/이미지 생성/챗봇" 같은 기능을 시트상 카테고리가 다른 도구(예: 옥수수)에 귀속시키면 → 해당 문장에 ⚠️ 표시 후 그대로 반환(차단보다는 가시화).
- 기본 라이브러리만 사용(Zod, 정규식). 외부 의존성 없음.

### 4. UI 신뢰도 표시

`src/components/consult/Dashboard.tsx`에서:
- 처방전 카드 헤더에 "AI 생성 초안 · 현장 검증 필수" 배지 추가.
- `assumptions` 섹션을 접을 수 있는 패널로 별도 노출 ("AI가 가정한 도구·단원 정보 보기").
- ⚠️ 표시된 문장은 amber 톤으로 강조해 교사가 즉시 식별 가능하게.

## 변경 범위

| 파일 | 변경 |
|---|---|
| `src/lib/tool-factsheet.ts` | 신설 — 도구 사실 카드 + 조회 함수 |
| `src/lib/prescription.functions.ts` | 시스템 프롬프트 강화, 동적 factsheet 삽입, `assumptions` 필드 추가, 사후 검증 |
| `src/components/consult/Dashboard.tsx` | 신뢰도 배지, assumptions 섹션, ⚠️ 강조 |

## 영향 범위 (변경 없음)

- 진단 분류 로직(`classify.ts`), 설문 흐름, 인증, 데이터베이스 스키마는 건드리지 않음.
- 모델은 일단 `gemini-2.5-flash` 유지 (성능 부족이 확인되면 별도 결정으로 `gemini-2.5-pro`/`gpt-5` 승격).

## 한계 (사용자에게 미리 공유)

- Factsheet는 **수기로 관리하는 진실 자료**입니다. 새 도구가 등장하면 시트를 갱신해야 합니다 — 초기 시드 6~10개로 시작하는 것을 추천.
- LLM 할루시네이션을 0으로 만들 수는 없습니다. 본 계획은 ① 자주 쓰는 도구의 가짜 기능 주장 차단, ② 불확실성을 UI에서 가시화, 두 가지를 목표로 합니다.
